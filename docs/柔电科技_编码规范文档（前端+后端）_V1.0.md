# 柔电科技 编码规范文档（前端+后端） V1.0

## 一、通用规范

### 1.1 命名规范

| 对象 | 规范 | 示例 |
|------|------|------|
| 文件 | kebab-case | `goal-service.ts`, `feedback-form.tsx` |
| 目录 | kebab-case 或 复数 | `modules/goal/`, `components/` |
| 类 | PascalCase | `GoalService`, `DecisionCaseAggregate` |
| 函数/方法 | camelCase | `createGoal()`, `findByOwner()` |
| 变量 | camelCase | `goalId`, `userName` |
| 常量 | UPPER_SNAKE_CASE | `MAX_RETRY_COUNT`, `DEFAULT_PAGE_SIZE` |
| 数据库表 | snake_case 复数 | `goals`, `decision_cases` |
| 数据库字段 | snake_case | `owner_id`, `created_at` |
| API 路径 | kebab-case 复数 | `/api/goals`, `/api/decision-cases` |

### 1.2 注释规范

- 公共 API / exported 函数：JSDoc 必写
- 复杂业务逻辑：行内注释解释 Why（不是 What）
- TODO/FIXME/HACK：标注负责人和日期
- 不要的代码直接删除，不要注释掉留着

### 1.3 文件组织

- 单文件 ≤ 400 行，超标即拆分
- 单函数 ≤ 50 行
- 单参数列表 ≤ 4 个参数（超过用对象）
- 嵌套层级 ≤ 3 层

## 二、前端规范（React + Next.js + TypeScript）

### 2.1 组件规范

```typescript
// ✅ 函数组件 + TypeScript
interface GoalCardProps {
  goal: Goal;
  onStatusChange: (id: string, status: GoalStatus) => void;
}

export const GoalCard: React.FC<GoalCardProps> = ({ goal, onStatusChange }) => {
  // ...
};
```

- 优先函数组件 + Hooks，不用 Class 组件
- 组件文件一行 export default
- Props 必须定义 interface/type
- 避免 any，善用泛型

### 2.2 状态管理

| 类型 | 工具 | 使用场景 |
|------|------|---------|
| 服务端状态 | React Query | API 请求、缓存、重取 |
| 全局 UI 状态 | Zustand | 侧边栏、当前组织、主题 |
| 表单状态 | React Hook Form + Zod | 所有表单 |
| 组件本地状态 | useState / useReducer | 仅本组件使用的状态 |

### 2.3 样式

- 使用 Tailwind CSS 或 Design Token (CSS Variables)
- 深蓝三色主题变量：`var(--navy-900)`, `var(--cyan-500)` 等
- 间距用 4 的倍数 (4px / 8px / 16px / 24px / 32px)
- 不超过 2 个强调色（navy + cyan）

### 2.4 Hooks 规范

- 自定义 Hook 以 `use` 开头
- useEffect 必须有依赖数组，禁止空依赖数组掩盖问题
- 避免在 useEffect 中做同步 state 更新（用 useMemo / 事件处理）

## 三、后端规范（NestJS + TypeScript + DDD）

### 3.1 模块结构

```
modules/goal/
├── goal.controller.ts      # HTTP 层
├── goal.module.ts
├── dto/
│   ├── create-goal.dto.ts
│   └── update-goal.dto.ts
├── application/
│   └── create-goal.usecase.ts
├── domain/
│   ├── goal.aggregate.ts
│   └── goal.entity.ts
├── infrastructure/
│   └── postgres-goal.repository.ts
```

### 3.2 DDD 分层原则

- Controller：只处理 HTTP，不做业务逻辑
- Application/UseCase：编排领域对象，一个 UseCase 一个 public 方法
- Domain：所有业务规则、不变量、状态机
- Infrastructure：数据库实现、外部 API 调用

### 3.3 异常处理

```typescript
// 使用 NestJS 内置异常
throw new NotFoundException('目标不存在');
throw new BadRequestException('状态流转不合法');
throw new ForbiddenException('无权限操作此资源');

// 全局异常过滤器统一格式
@Catch()
export class GlobalExceptionFilter implements ExceptionFilter {
  catch(exception: unknown, host: ArgumentsHost) {
    // 统一返回 { success: false, error: { code, message } }
  }
}
```

### 3.4 验证

- 所有 DTO 使用 class-validator 装饰器
- 复杂业务规则验证放在 Domain 层

### 3.5 日志规范

- `logger.log()` — 正常业务流程节点
- `logger.warn()` — 可恢复的异常（Redis 缓存降级查 DB）
- `logger.error()` — 需要人工介入的异常（数据库连接失败）
- 禁止 `console.log()`

## 四、数据库规范

- 所有表必须含 `id` (UUID v4), `created_at` (TIMESTAMPTZ), `updated_at` (TIMESTAMPTZ)
- 软删除表加 `deleted_at` (TIMESTAMPTZ, nullable)
- 外键必须定义（保证引用完整性）
- JSONB 字段仅用于 schema-less 数据（如 permission_set, payload），不滥用
- 禁止 `SELECT *`，明确列出需要的字段
