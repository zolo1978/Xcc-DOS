### 1\.2 设计目标

遵循字节跳动数据库开发规范，围绕**多租户强隔离、高并发读写、低延迟、可扩展、可审计、合规可控**核心目标设计。适配系统规则调度、会话管理、自进化反哺、运维管控、日志追溯全业务场景，支撑公网集群、内网单机、边缘离线三种部署模式，保障数据一致性、安全性、可运维性。

### 1\.3 设计约束与规范

- **数据库选型**：生产环境优先 MySQL 8\.0 / PostgreSQL 14\+，InnoDB 存储引擎，UTF8MB4 字符集，支持所有emoji及特殊字符存储

- **通用字段规范**：所有业务表强制携带 tenant\_id、create\_time、update\_time、delete\_flag 四大通用字段，用于租户隔离、数据溯源、逻辑删除

- **主键规范**：统一采用自增主键\+雪花ID混合模式，核心业务表使用雪花ID，避免自增ID暴露业务量级

- **索引规范**：高频查询场景建立联合索引，杜绝冗余索引、无效索引，禁止大字段建立索引

- **删除规范**：全部采用逻辑删除，禁止物理删除业务数据，保障数据可追溯、可恢复

- **脱敏规范**：隐私数据（手机号、身份证、地址等）数据库层面脱敏存储，无明文留存

- **字段命名**：小写蛇形命名，语义清晰、无歧义，所有字段、表名必须配备详细注释

## 二、整体数据架构设计

### 2\.1 数据分层架构

整体分为四层数据存储体系，各司其职、解耦存储，避免单表膨胀、冷热数据混杂

1. **核心业务数据层**：租户、用户、权限、规则、同义词、会话核心业务数据，高频读写、高一致性要求

2. **日志行为数据层**：请求日志、兜底日志、操作审计日志，海量写入、低频查询

3. **自进化任务数据层**：聚类任务、规则生成记录、版本快照、审核记录，定时任务驱动

4. **缓存临时数据层**：Redis 承载会话状态、热点规则、限流计数、临时缓存数据，提升并发性能

### 2\.2 多租户数据隔离方案

兼容双隔离模式，一键切换，适配不同合规与运营场景

- **共享库行级隔离（默认）**：所有租户共用一套数据库实例，通过 tenant\_id 字段做行级隔离，低成本、易运维，适配中小规模SaaS场景

- **独立物理库隔离（高阶）**：单租户独立数据库实例，完全物理隔离，满足金融、政务等高合规、高安全场景

全链路强制租户条件拦截，所有CRUD操作必须携带 tenant\_id，杜绝跨租户数据串扰、越权查询。

### 2\.3 数据生命周期规范

- **实时业务数据**：永久保留，仅逻辑删除，支持手动恢复

- **实时请求日志**：保留7天，到期自动归档至冷存储

- **归档日志数据**：冷存储保留90天，到期自动物理清理

- **规则版本快照**：单租户最大保留50份，超出自动清理最旧版本

- **过期会话数据**：会话超时24小时后自动物理清理，释放存储资源

## 六、数据库回滚降级预案

本章针对Prolog AgentTeam智能交互工厂数据库投产运行、版本迭代、线上异常场景，制定标准化**降级、熔断、回滚、故障恢复**全套预案。覆盖数据库变更失误、性能雪崩、数据异常、租户隔离失效、高并发故障等核心风险场景，明确故障分级、触发条件、执行步骤、恢复标准，保障线上业务高可用、数据安全可控、故障快速止血，适配公网集群、内网单机、边缘离线全部署模式。

### 6\.1 预案总则

#### 6\.1\.1 适用范围

本预案适用于系统V1\.2及后续迭代版本，所有数据库相关变更、线上故障、性能异常、数据错乱、权限越权、租户数据泄露、日志异常、自进化任务失败等场景的应急处理，涵盖开发、测试、投产、运维全生命周期。

#### 6\.1\.2 核心原则

- **优先止血**：故障发生后优先执行降级、熔断、流量拦截，快速恢复业务可用，再定位修复问题，避免故障扩大。

- **数据优先**：所有回滚、降级操作以**数据不丢失、数据不乱序、租户不串扰**为核心前提，禁止任何风险操作导致数据损坏。

- **可追溯可恢复**：所有应急操作留痕记录，支持二次回滚、数据修复、版本追溯，保障故障闭环。

- **分级处置**：根据故障影响范围、严重程度分级处理，精准匹配降级、回滚、停机修复等处置方案。

- **合规可控**：所有操作符合数据脱敏、逻辑删除、租户隔离、数据留存规范，无合规漏洞。

#### 6\.1\.3 前置依赖

常态化保障数据库定时备份、版本快照留存、日志完整归档、索引配置合规、缓存预热正常，为应急回滚降级提供基础支撑，无备份、无快照禁止执行高危变更。

### 6\.2 故障分级标准

结合业务影响范围、数据安全风险、服务可用度，将数据库故障划分为三级，对应不同处置策略：

|故障等级|故障定义与影响范围|核心处置策略|响应时效|
|---|---|---|---|
|**P1 重大故障**|全平台服务不可用、大面积租户数据错乱/泄露、数据库宕机、核心表损坏、跨租户数据串扰、数据批量丢失|立即全量降级、暂停所有业务写入、紧急回滚版本、冻结数据库变更、人工介入抢修|5分钟内响应，30分钟内完成止血|
|**P2 严重故障**|部分租户服务异常、单表读写失败、高并发卡顿、规则失效、会话异常、日志写入中断，无数据安全风险|局部降级、限流熔断、隔离异常租户/接口、增量回滚、优化数据库性能|10分钟内响应，1小时内恢复业务|
|**P3 一般故障**|个别功能异常、少量数据延迟、非核心日志缺失、页面展示异常，核心业务正常可用|功能降级、临时规避、后台修复、无需全量回滚，迭代修复漏洞|30分钟内响应，24小时内完成修复|

### 6\.3 通用降级策略（常态化兜底）

系统运行期间触发数据库压力过高、接口超时、并发溢出等预警时，自动触发通用降级策略，保障核心业务稳定运行，所有降级策略均为无损降级，不影响核心数据安全。

#### 6\.3\.1 性能降级

- **读写分离降级**：数据库CPU、内存、磁盘IO持续超过80%阈值时，自动开启读写分离，读请求分流至从库，主库仅承担写入操作，降低主库压力。

- **并发限流降级**：单租户/全平台请求QPS超过阈值时，自动拦截高频无效请求，限制非核心接口并发，优先保障规则匹配、会话交互等核心能力。

- **缓存兜底降级**：数据库查询超时、短暂不可用时，自动启用Redis热点缓存数据兜底，返回缓存内的规则、同义词、租户基础数据，保障用户正常交互。

#### 6\.3\.2 功能降级

- **自进化任务降级**：数据库负载过高时，自动暂停聚类任务、规则自动生成、版本快照批量生成等非实时任务，待负载恢复后自动续跑。

- **日志归档降级**：高并发峰值时段，暂停日志冷热迁移、过期日志清理任务，优先保障实时日志写入与核心业务读写。

- **灰度发布降级**：检测到数据库异常时，自动终止所有规则灰度放量，灰度比例归零，仅保留稳定版本规则对外提供服务。

#### 6\.3\.3 租户隔离降级

- 检测到单租户异常高频请求、恶意刷量导致数据库压力异常时，自动隔离该租户，限制其写入操作，其他租户业务不受影响。

- 物理隔离租户出现数据库异常时，自动冻结该租户独立库操作，避免异常扩散至全局集群。

### 6\.4 核心场景回滚预案

针对数据库版本变更、表结构修改、数据更新、规则迭代等高风险操作，制定精准回滚方案，所有回滚操作均支持一键执行、快速恢复。

#### 6\.4\.1 数据库表结构变更回滚

**触发场景**：新增/修改字段、调整索引、修改字段类型长度后，出现读写报错、数据截断、索引失效、业务适配异常等问题。

**前置条件**：所有表结构变更前已备份原表结构与存量数据，留存变更SQL脚本。

**执行步骤**：

1. 立即暂停对应表的业务写入操作，接口临时降级只读，避免异常数据新增。

2. 执行反向回滚SQL，恢复原表字段、数据类型、长度、索引结构，删除新增无效字段与冗余索引。

3. 校验回滚后表结构与基线版本一致，核对通用字段、主键、唯一索引有效性。

4. 恢复业务读写，抽样校验数据完整性、查询功能、关联逻辑正常。

**恢复标准**：表结构匹配设计规范，无报错、无数据错乱，所有业务功能正常使用。

#### 6\.4\.2 规则版本迭代回滚

**触发场景**：规则新增、编辑、灰度发布后，出现意图匹配失败、流程报错、业务逻辑异常、性能下降等问题。

**前置条件**：系统自动留存所有规则历史快照版本，支持版本追溯与回滚。

**执行步骤**：

1. 紧急下线异常规则版本，将规则状态改为未生效，暂停灰度放量。

2. 通过rule\_snapshot快照表，回滚至上一个稳定可用版本，恢复规则源码与配置参数。

3. 锁定当前规则版本，禁止自动迭代更新，规避二次异常。

4. 校验规则匹配、流程调度功能正常，核对租户数据隔离有效性。

5. 问题修复后，重新迭代发布新版本，完成灰度验证后放量。

**恢复标准**：规则运行正常，业务逻辑无误，无报错、无性能异常，租户隔离正常。

#### 6\.4\.3 批量数据更新回滚

**触发场景**：批量更新租户、用户、同义词数据后，出现数据错误、状态异常、隐私数据泄露风险等问题。

**前置条件**：批量操作前已完成全量数据备份，留存操作日志。

**执行步骤**：

1. 立即暂停所有批量数据操作，冻结对应数据表写入权限。

2. 基于备份数据还原异常时段数据，覆盖错误更新内容。

3. 校验通用字段、脱敏数据、租户ID、删除标识合规无误。

4. 排查批量脚本漏洞，修复问题后重新执行合规更新。

5. 开放数据写入权限，恢复正常业务流程。

**恢复标准**：数据完全恢复至操作前状态，无错误数据、无隐私泄露，业务正常运行。

#### 6\.4\.4 租户隔离故障回滚

**触发场景**：出现跨租户数据查询、数据串扰、越权访问等租户隔离失效问题。

**执行步骤**：

1. 紧急熔断所有租户数据查询接口，禁止跨租户数据访问。

2. 校验全链路租户ID拦截逻辑，修复缺失租户条件的代码与SQL。

3. 批量校验所有业务数据租户归属，修正错乱数据。

4. 恢复接口访问权限，逐租户验证数据隔离有效性。

5. 物理隔离模式异常时，立即切换为行级共享模式兜底，保障业务可用。

**恢复标准**：租户数据完全隔离，无跨租户串扰、无越权查询，隔离模式正常生效。

### 6\.5 重大故障应急恢复预案（P1级）

#### 6\.5\.1 数据库宕机/核心表损坏

1. **紧急止血**：立即切换备用数据库节点，启用离线缓存兜底，暂停所有业务写入，避免数据丢失。

2. **故障排查**：快速核查磁盘损坏、进程异常、锁表、SQL死锁、索引损坏等根因。

3. **数据恢复**：基于最近一次全量备份\+binlog日志，恢复核心业务表数据，保证数据零丢失。

4. **业务恢复**：恢复数据库读写能力，逐表校验数据完整性、索引有效性、租户隔离能力。

5. **复盘加固**：修复底层故障问题，优化数据库配置，增加监控告警，规避复现。

#### 6\.5\.2 批量数据泄露/损坏

1. **紧急阻断**：立即关闭数据查询、导出接口，冻结所有租户数据，阻断泄露通道。

2. **数据隔离**：隔离异常数据批次，区分正常数据与异常数据，避免污染扩散。

3. **数据修复**：通过备份快照还原损坏数据，重新脱敏隐私数据，清理错误数据。

4. **合规校验**：全量校验数据脱敏、逻辑删除、租户隔离合规性，消除合规风险。

5. **恢复上线**：确认无数据安全风险后，逐步恢复业务访问，持续监控异常。

### 6\.6 故障恢复校验标准

所有降级、回滚操作完成后，必须完成以下校验，方可解除应急状态、恢复全量业务：

- **数据完整性**：核心业务数据无丢失、无错乱、无重复，历史版本快照完整可追溯。

- **功能可用性**：所有核心业务功能正常，规则匹配、会话交互、租户管理、用户权限无异常。

- **隔离安全性**：多租户隔离正常，无跨租户串扰、越权访问问题。

- **性能稳定性**：数据库读写延迟正常，无锁等待、无全表扫描、无并发雪崩，索引全部生效。

- **合规规范性**：通用字段完整、数据脱敏有效、逻辑删除正常、字符集无乱码，符合设计规范。
