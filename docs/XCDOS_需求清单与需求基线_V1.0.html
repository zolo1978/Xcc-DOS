二、版本迭代信息

迭代编号

XCDOS-MVP-V1.0

需求来源

内部产品规划 + 柔电科技管理层需求调研

业务背景

企业决策缺乏结构化工具，目标到执行链路断裂，老板看不到业务全貌。亟需一套"目标→问题→决策方案→任务→反馈→看板"的闭环系统。

价值收益

目标执行可追踪、决策过程可审计、异常可预警，预期减少 30% 管理沟通成本

四、MVP 范围定义

4.1 范围内（In Scope）

核心闭环

目标 → 问题 → 决策方案 → 任务 → 反馈 → 看板

用户角色

老板 / 高管 / 中层管理者 / 员工 / 系统管理员 共 5 角色

Agent 辅助

Agent 辅助拆解、评估、报告生成，所有输出可审计

数据看板

老板驾驶舱含 KPI/异常/进度 3 大面板

权限体系

RBAC 五级权限，按角色控制页面/操作/数据范围

审计日志

全量操作留痕，支持追溯

4.2 范围外（Out of Scope）

  • 不做完整微服务拆分（MVP 采用模块化单体 + DDD 分层）
  • 不替代 ERP / MES / WMS / CRM 等外部业务系统
  • 不允许 Agent 自动执行高风险写操作（删除/批量修改需人工审批）
  • 不做复杂 BI 分析平台（仅做核心驾驶舱）
  • 不做移动端原生 App（MVP 仅 Web Admin + H5）
  • 不做外部系统 API 集成对接（MVP 后二期规划）
  • 不做多语言/国际化

七、需求基线锁定

7.1 锁定信息

属性内容
基线锁定日期MVP 需求评审通过当日
锁定范围本文档第三~四章节所列全部功能
锁定人产品负责人

7.2 锁定原则

  1. 需求基线一旦锁定，任何功能性新增/修改/删除须走变更流程（CR）
  2. P0 功能变更须技术负责人 + 产品负责人双审批
  3. P1 / P2 功能变更由产品负责人审批
  4. 所有变更必须更新本文档第五章变更记录并留痕
  5. 变更影响评估需包含：工时影响、测试回归范围、上线风险

7.3 锁定后允许的直接调整（不走变更流程）

  • UI 文案优化（不改变交互逻辑与功能行为）
  • 错误提示语调整
  • 非功能性的性能优化（不改变 API 契约）
  • Bug 修复（对齐 PRD 原意，非功能变更）
