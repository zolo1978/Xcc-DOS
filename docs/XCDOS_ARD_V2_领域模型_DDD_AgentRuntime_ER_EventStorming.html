A. 当前认知状态

先把不确定性摆出来，免得人类又把“老板一句话”误认为“完整需求”。当前文档是架构假设 V2.0，不是终局定义。

产品核心是“拆-推-评-算 + 定标准追过程拿结果 + 强制反馈 + 实时数据”。

一、架构目标与边界

本 ARD 的目标是把产品从“说明书”推进到“可研发拆解”的架构层。架构不是装饰，是为了防止项目长成一坨会登录的 Excel。

目标 1：统一决策流程

目标/问题进入拆推评算，形成可追溯决策链。

目标 2：统一执行闭环

标准、任务、过程、反馈、结果统一建模。

目标 3：统一 Agent 运行底座

业务 Agent 不各自乱飞，统一由 Runtime 调度。

系统边界

边界项包含暂不包含
决策拆解、预测、可行性评估、ROI 测算、决策报告完全替代老板决策
执行标准、任务、过程、反馈、异常、结果复杂 ERP/MES 全量替代
AgentDecision / HR / PMC / Sales / Lean / Report Agent无边界自动执行生产或财务动作
数据目标、任务、反馈、结果、事件、指标一次性接入所有历史系统脏数据

二、领域模型 Domain Model

XCDOS 的核心领域不是“人事”或“生产”，而是“企业目标如何被拆解、推演、评估、测算、执行、反馈和复盘”。

三、DDD 架构设计

按领域驱动设计拆分边界上下文，避免 HR、PMC、销售、精益模块互相污染。系统不怕大，怕的是边界不清，大锅炖软件通常最后只有锅活着。

4.2 Agent 权限分级

级别能力示例是否需要人工确认
L1 只读分析查询、总结、分析生成日报摘要否
L2 建议生成生成方案、评分、测算输出拆推评算报告否/按配置
L3 工作流变更创建任务、调整节点、提醒人员创建异常处理任务部分需要
L4 业务系统写入写入 ERP/MES/CRM 等外部系统变更排产、调整订单需要
L5 高风险动作财务、人事处分、生产停线冻结订单、处罚员工必须人工确认

八、权限、安全与审计架构

企业管理系统里的 Agent 如果没有权限边界，就是一个会写报告的安全事故。当前阶段更建议“人机协同审批”，不要让 Agent 一上来就拥有改变世界的幻觉。

主体权限范围关键限制
老板全局看板、决策审批、异常处理高风险操作仍需二次确认
总监部门目标、任务、反馈、异常不可查看非授权部门敏感数据
经理团队目标、任务分配、反馈审核不可修改历史反馈记录
员工个人任务、反馈、结果不可删除已提交反馈
Agent按授权执行查询、建议、任务创建L4/L5 操作必须人工确认

审计要求

  • 所有 Agent Run 必须记录：输入、上下文摘要、工具调用、输出、执行人/触发源。
  • 所有关键业务写入必须生成 DomainEvent。
  • 所有反馈修改必须保留版本历史。
  • 所有高风险操作必须支持回放与追责。

九、MVP 技术落地路径

阶段性更建议先跑通“目标 → 拆推评算 → 任务 → 反馈 → 看板”主链路。别一口吃成平台，平台通常是被小闭环喂大的，不是 PPT 许愿长大的。

十、风险、反例与最小验证动作

这里是专门防止“看起来很强，落地一地鸡毛”的部分。文明的进步，偶尔也需要承认系统会翻车。

风险可能反例验证动作修正策略
用户不愿意反馈强制反馈变成形式主义日报选销售/PMC 试点 2 周，看反馈质量反馈结构最小化，和任务/异常强绑定
数据源质量差预测和评分建立在脏数据上先用人工录入 + 表格导入验证建立数据可信度评分
Agent 输出不可信方案看起来合理但依据不足所有输出要求证据、反例、置信度引入规则引擎和人工审批
系统边界过大想替代 ERP/MES/OA，最后都没做好MVP 只做主链路闭环外部系统只做接入，不做替代
管理层不参与只要求员工反馈，管理层不进入系统试点要求管理层同结构反馈老板驾驶舱暴露管理层反馈率

阶段性判断

当前较强判断：XCDOS 的 V2.0 架构主路径应以 Decision Context + Execution Context + Feedback Context + Agent Runtime 四个核心上下文为主，不宜先做成大而全 ERP 替代品。

最小验证动作

先选择一个真实部门，例如销售或 PMC，用 2 周跑通：创建目标 → 拆解问题 → 生成 3 个方案 → 评估可行性 → 测算 ROI → 生成任务 → 每日反馈 → 老板驾驶舱查看异常。成立后再扩展 HR、生产、精益模块。
