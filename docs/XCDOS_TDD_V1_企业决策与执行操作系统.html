XCDOS 企业决策与执行操作系统
TDD V1.0

基于 PRD V1.0 与 ARD V2.0，输出可研发落地的技术设计方案，覆盖前端、后端、数据库、API、Agent Runtime、工作流、权限、安全、部署、测试与开发计划。

架构范式：DDD + CQRS Lite 前端：React / Next.js 后端：NestJS / PostgreSQL Agent：Runtime + Tool Router + Trace

A. 当前认知状态

已知事实

MVP 需要跑通目标、拆推评算、任务、反馈、看板主链路。

当前推测

采用模块化单体优先，后续再拆微服务，阶段性更稳。

缺失信息

企业实际组织层级、外部系统接口、部署环境、账号体系尚待确认。

不能下定论原因

Agent 自动执行边界、数据质量、试点部门流程仍需验证。

一、技术目标与非目标

目标 1：主链路闭环

目标 → 问题 → 决策方案 → 任务 → 反馈 → 看板。

目标 2：Agent 可审计

所有 Agent 输出、工具调用、上下文摘要必须可追踪。

目标 3：可扩展

支持后续接入 HR、PMC、Sales、Lean、ERP/MES/CRM。

非目标

  • MVP 不做完整微服务拆分。
  • MVP 不替代 ERP / MES / WMS / CRM。
  • MVP 不允许 Agent 自动执行高风险写操作。
  • MVP 不做复杂 BI 平台，只做核心驾驶舱。

十、权限、安全与审计

10.1 RBAC 权限模型

角色权限范围限制
boss全局读、审批、异常处理高风险二次确认
admin系统配置、用户、角色、日志不代替业务审批
manager部门目标、任务、反馈、异常不可越权部门
employee个人任务、反馈、通知不可删除反馈
agent按授权工具调用L4/L5 需要人工确认

10.2 审计日志

  • 所有写操作记录 actor_id、action、resource_type、resource_id。
  • 关键字段变更记录 before_value 与 after_value。
  • Agent 输出记录 input_summary、context_summary、tool_calls、output_summary。
  • 反馈提交后不可物理删除，只能作废并保留历史。

十三、开发计划与任务拆解

阶段周期后端任务前端任务验收

Sprint 51 周优化、测试、部署体验修复、响应式、演示数据试点上线

阶段性判断

当前 TDD 建议先以 8-10 周完成 MVP：主链路闭环 + 驾驶舱 + 拆推评算 + 基础 Agent Trace。外部系统对接和完整 HR/PMC Agent 放到二期，不然项目会像每个伟大系统一样，死在“顺便也做一下”。
