XCDOS API 接口文档
V1.0

基于 TDD V1.0 第七章与 ARD V2.0 第七章，定义 XCDOS 全部 REST API 端点的请求/响应契约、认证方式、限流规则、错误码体系。前后端联调唯一依据。

端点总数：28 Base URL：/api Content-Type：application/json 协议：HTTPS only

一、API 约定

项规范
Base URL/api（生产：https://{domain}/api）
Content-Typeapplication/json; charset=utf-8
字符编码UTF-8
认证方式Bearer Token (JWT)，Header: Authorization: Bearer <token>
追踪 IDHeader: X-Trace-Id（客户端传入或网关自动生成 UUID）
限流默认 100 req/min per user；登录接口 10 req/min per IP
超时默认 30s；Agent Run 接口 120s
重试策略幂等 GET 请求可重试 3 次（间隔 1s/2s/4s）；写操作不自动重试
时间格式ISO 8601 (2026-06-11T14:30:00+08:00)
分页Query: ?page=1&pageSize=20；Response: { items, total, page, pageSize }

统一响应信封

{
  "success": true,
  "data": { ... },
  "error": null,
  "traceId": "req_a1b2c3d4",
  "timestamp": "2026-06-11T14:30:00+08:00"
}

// 错误时
{
  "success": false,
  "data": null,
  "error": { "code": "GOAL_NOT_FOUND", "message": "目标不存在或已删除" },
  "traceId": "req_a1b2c3d4",
  "timestamp": "2026-06-11T14:30:00+08:00"
}

二、认证接口

POST/api/auth/login

用户登录，返回 JWT Token。限流：10 req/min/IP。

参数位置类型必填说明
emailBodystring✓登录邮箱
passwordBodystring✓密码（明文，HTTPS 传输加密）

Response 200: { success: true, data: { token, user: { id, name, email, role } } }

Error 401: AUTH_INVALID_CREDENTIALS

GET/api/me

获取当前登录用户信息。

Response 200: { id, name, email, phone, role: { id, name, level, permission_set }, org: { id, name } }

四、目标管理

POST/api/goals

创建目标。权限：manager+。

参数类型必填说明
namestring✓目标名称，1-200 字符
descriptionstring目标描述
ownerIdUUID✓目标负责人
metricTypeenum✓amount / count / percentage / boolean
targetValuenumber✓目标值
startDatedate✓开始日期
deadlinedate✓截止日期
parentIdUUID父目标 ID（拆解时使用）
orgIdUUID✓所属组织

Response 201: { id, name, status: "draft", ... }

GET/api/goals

目标列表（分页）。Query: ?status=active&ownerId=xxx&orgId=xxx&page=1&pageSize=20。权限：manager+ 见全组织，employee 仅见自己相关。

GET/api/goals/:id

目标详情，含子目标列表、关联任务/问题摘要。

POST/api/goals/:id/breakdown

目标拆解——创建子目标。Body 同 create goal（parentId 自动设置为 :id）。

PATCH/api/goals/:id/status

更新目标状态。Body: { status: "active" | "completed" | "cancelled" }。请求头 If-Match: <version>（乐观锁，版本不匹配返回 409 VERSION_CONFLICT）。权限：目标 owner 或 manager+。

DELETE/api/goals/:id

软删除目标（写 deleted_at，不物理删除）。前置校验：目标下存在未完成任务（todo / in_progress）时返回 409 GOAL_HAS_ACTIVE_TASKS。权限：目标 owner 或 manager+。Response 204。

创建问题。权限：manager+。

参数类型必填说明
goalIdUUID✓关联目标
titlestring✓问题标题
descriptionstring问题描述

六、决策案例（拆推评算）

POST/api/decision-cases

创建决策案例。Body: { problemId, title, stage: "dismantle" }。权限：manager+。

GET/api/decision-cases/:id

决策案例详情，含假设列表、评估分数、ROI、方案。聚合视图。

POST/api/decision-cases/:id/hypotheses

添加假设。Body: { content, evidenceScore, confidence, counterExample }。案例 stage 变为 dismantle。

POST/api/decision-cases/:id/forecast

推演阶段。基于假设生成推演并持久化到 forecasts 表（P0-03）。Body: { selectedHypothesisIds: [...] }。Response 201: { id, caseId, version, scenarios: [{ name, probability, outcome, impact, assumptions }], confidence, modelSource, agentRunId }。重复调用生成新 version，不覆盖历史。案例 stage → hypothesize。

GET/api/decision-cases/:id/forecasts

推演版本列表（含人工修订记录）。Query: ?version= 取指定版本。决策报告引用的 forecast version 由此接口可追溯（P0-03 审计闭环）。

POST/api/decision-cases/:id/evaluate

多维评估。Body: { resourceScore, timeScore, riskScore, feasibilityScore, comment }。案例 stage → evaluate。

POST/api/decision-cases/:id/simulate-roi

ROI 试算。Body: { cost, revenue, assumptions: {...} }。返回 { roi, paybackDays }。案例 stage → calculate。

POST/api/decision-cases/:id/report

生成决策报告。自动汇总拆推评算全流程数据。案例 stage → report。注意：报告生成不改变 status；案例 status → resolved 的前置条件 = 该案例下至少一个 Plan 处于 approved 状态（P0-02 主链路闭环）。

六.B 方案（Plan）管理与审批

状态机：draft → submitted → approved | rejected；approved 后随任务执行进入 executing → completed。rejected 可修改后重新 submit。只有 approved 的 Plan 允许被 POST /api/tasks 引用（planId 前置校验）。

POST/api/decision-cases/:id/plans

基于决策案例创建执行方案（status = draft）。Body: { title, description }。权限：案例 owner 或 manager+。Response 201: { id, caseId, status: "draft" }

GET/api/plans/:id

方案详情，含状态、审批人、审批时间、驳回原因、关联任务数。

POST/api/plans/:id/submit

提交审批：draft | rejected → submitted。仅方案 owner。写入 submitted_at；非法状态返回 409 INVALID_PLAN_STATE。

POST/api/plans/:id/approve

审批通过：submitted → approved。权限：manager+ 且不得为方案 owner 本人（职责分离）。写入 approved_by / approved_at，记审计事件 PlanApproved。

POST/api/plans/:id/reject

驳回：submitted → rejected。Body: { reason }（必填）。权限同 approve。记审计事件 PlanRejected。

七、任务管理

POST/api/tasks
参数类型必填说明
planIdUUID✓关联执行方案；服务端校验 Plan.status 必须为 approved，否则 422 PLAN_NOT_APPROVED（P0-02）
goalIdUUID关联目标
ownerIdUUID✓负责人
titlestring✓任务标题
descriptionstring任务描述
dueTimedatetime✓截止时间
standardIdUUID关联执行标准
GET/api/tasks

任务列表。Query: ?ownerId=&status=&goalId=&page=&pageSize=。权限：manager+ 见全组织，employee 仅见自己的。

PATCH/api/tasks/:id/status

更新任务状态。Body: { status: "todo" | "in_progress" | "done" }。

状态机规则：todo→in_progress→done；done 后不可再改（需管理员重置）。超过 dueTime 未 done 自动 delayed。

八、反馈管理

POST/api/tasks/:id/feedback

提交任务反馈。同一任务同日（business_date）已有生效反馈时返回 409 DUPLICATE_FEEDBACK，不覆盖原文；修改须走修订接口（见下）。请求须携带 Idempotency-Key 头防重试重复。

参数类型必填说明
todayGoalstring✓今日目标
resultstring✓今日结果
blockerstring✓卡点 / 阻塞
nextActionstring✓下一步行动

Response 201: { id, taskId, revision: 1, qualityScore, submittedAt }

qualityScore 由 Agent 异步评估（0-100），初始为 null。

POST/api/feedbacks/:id/revisions

对已提交反馈创建修订（P0-04：不可篡改，原文保留）。生成新行 revision+1，原行 superseded_by 指向新行。仅反馈提交人本人可修订；当日 23:59 后锁定不可修订。Body 同提交接口四字段。

Response 201: { id, revision, supersedes, submittedAt }

Errors: 403 非本人；409 已被更新修订替代；423 已锁定。

九、异常管理

GET/api/exceptions

异常列表。Query: ?severity=&status=&ownerId=&page=&pageSize=。权限：manager+ 见全组织，employee 见自己被指派的。

PATCH/api/exceptions/:id/status

更新异常状态。Body: { status: "acknowledged" | "resolving" | "resolved" }。权限：异常 owner 或 manager+。

十、Agent Run 追踪

GET/api/agent-runs

Agent 运行记录列表。Query: ?agentType=&status=&page=&pageSize=。权限：admin。

POST/api/agent-runs/:id/cancel

取消运行中的 Agent Run：status running → cancelled，幂等（已终态返回 200 原状态）。中断后已产生的工具调用记录保留，审计不回滚。权限：发起人或 admin。

GET/api/agent-runs/:id/trace

Agent Run 完整追踪。权限：admin。
