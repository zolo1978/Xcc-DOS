XCDOS 企业决策与执行操作系统
PRD V1.0

基于产品说明书与 ARD V2.0，面向研发、设计、测试、交付输出可执行产品需求。核心目标：跑通“目标/问题 → 拆推评算 → 决策方案 → 执行任务 → 强制反馈 → 数据看板”的 MVP 闭环。

产品定位：企业决策与执行操作系统 核心模型：拆-推-评-算 MVP：销售/PMC 试点 UI：深蓝三色专业版

A. 当前认知状态

已知事实

产品核心由拆推评算、定标准追过程拿结果、强制反馈、实时数据构成。

当前推测

MVP 更适合先从销售或 PMC 单部门切入，而非全公司一次性铺开。

缺失信息

真实组织架构、岗位清单、当前数据源、反馈制度、老板审批规则尚未完全确认。

不能下定论原因

强制反馈机制是否被接受，取决于老板推动力度和管理层是否同标准执行。

PRD 主假设池

假设证据强度解释力反例抗压可验证性行动价值总分段位

H2：先做完整 Agent Runtime，平台能力更强222210101377A
H3：先做 HR Agent，老板最容易感知管理价值20199121272A
当前优先验证路径：以销售或 PMC 为试点，先跑通主链路闭环，再扩展 Agent 和外部系统。别一上来就造企业级宇宙，宇宙通常不按期交付。

一、产品概述

XCDOS 是芯诚科技面向物联网、RFID/NFC 卡片、生产、销售、PMC 与组织管理场景的企业决策与执行操作系统。

产品目标

把企业目标、问题、方案、任务、反馈、结果统一到一条可追踪流水线。

核心价值

让老板看见真实过程，让管理层按标准追过程，让员工强制反馈。

产品边界

不替代 ERP/MES/CRM，先作为决策与执行中枢，逐步接入外部系统。

一句话定位

XCDOS = 拆推评算做正确的事 + 定标准追过程拿结果把事做正确 + 强制反馈实时数据持续纠偏。

三、版本范围与优先级

3.1 MVP 范围

模块是否进入 MVP优先级说明
老板驾驶舱是P0展示目标、异常、反馈率、任务进度
目标管理是P0创建公司/部门/个人目标并拆解
拆推评算中心是P0目标/问题拆解、方案评估、ROI 测算
任务与过程追踪是P0任务分配、状态更新、节点跟踪
强制反馈是P0每日反馈、未反馈提醒、异常升级
基础 Agent是P1Report Agent / Reminder Agent / Decision Assistant
HR Agent 完整版否P2二期扩展人才画像、绩效成长
外部 ERP/MES/WMS 对接否P2MVP 可先支持 Excel 导入

3.2 不做范围

  • 不在 MVP 阶段替代 ERP、MES、WMS、CRM。
  • 不允许 Agent 自动执行高风险业务动作。
  • 不做复杂财务核算、人事处分、生产停线自动化。
  • 不做花里胡哨大屏，老板要的是判断依据，不是看烟花。

五、核心业务流程

5.1 目标到执行闭环

1. 创建目标

老板/负责人创建公司或部门目标。

→
2. 拆解目标

拆为子目标、任务、指标和责任人。

→
3. 生成任务

绑定标准、截止时间和负责人。

→
4. 强制反馈

每日提交目标、结果、卡点、下一步。

→
5. 看板更新

驾驶舱实时展示进度与异常。

5.2 问题到决策闭环

1. 发现问题

人工录入或系统从异常中生成。

→
2. 拆

拆解原因、变量、责任边界。

→
3. 推

推演多个方案的未来结果。

→
4. 评

评估资源、风险、成功率。

→
5. 算

测算投入产出与回本周期。

→
6. 决策

老板/负责人审批方案。

七、关键字段与业务规则

7.1 Goal 目标字段

字段类型必填说明
goal_idstring是目标唯一 ID
goal_namestring是目标名称
owner_idstring是负责人
metric_typeenum是金额/数量/百分比/次数/布尔
target_valuenumber是目标值
deadlinedatetime是截止时间
statusenum是未开始/进行中/完成/延期/取消

7.2 Feedback 反馈字段

字段类型必填规则
today_goaltext是不能少于 10 字
resulttext是必须描述完成结果
blockertext否无卡点可填“无”
next_actiontext是必须有下一步动作
submitted_atdatetime是自动生成

7.3 核心业务规则

  • 目标必须绑定负责人、指标、目标值、截止时间。
  • 任务必须绑定负责人和截止时间。
  • 反馈必须包含今日目标、今日结果、当前卡点、下一步动作。
  • 超过截止时间未反馈，系统自动生成 FeedbackMissed 事件。
  • P0/P1 异常必须进入老板驾驶舱。
  • Agent 输出的方案必须包含依据、反例、置信度、验证动作。

十、验收标准

10.1 功能验收

  • 可创建公司/部门/个人目标。
  • 目标可拆解为子目标和任务。
  • 可从目标或异常发起拆推评算。
  • 系统可生成至少 3 个候选方案。
  • 方案可进行可行性评分。
  • 方案可进行 ROI 测算。
  • 可生成决策报告。
  • 任务可分配负责人和截止时间。
  • 员工可提交每日反馈。
  • 未反馈可自动提醒。
  • P0/P1 异常进入老板驾驶舱。
  • 老板可查看目标、任务、反馈、异常总览。

10.2 质量验收

类别标准
性能核心页面首屏加载 ≤ 2.5s；列表查询 ≤ 1s
可用性普通员工提交反馈路径不超过 3 步
安全不同角色只能访问授权数据
审计关键写操作必须记录操作人、时间、变更前后内容
兼容支持 Chrome / Edge 最新两个主版本

十二、风险与验证计划

风险表现验证方式应对策略
反馈形式主义员工填空话套话抽样检查反馈质量增加反馈质量评分和重写提示
管理层不执行只要求员工反馈看管理层反馈率老板驾驶舱公开管理层反馈率
拆推评算太复杂用户不会用观察首次使用完成率提供向导式流程和模板
数据不准看板和现实不一致人工抽检 20 条数据建立数据来源和可信度标识
范围失控不断加 ERP/MES 功能版本需求评审MVP 坚持只做主链路

阶段性判断

当前阶段不建议把 XCDOS 做成大而全系统，而应先做“管理闭环最小可用系统”：目标、任务、反馈、异常、驾驶舱、拆推评算。成立后再加完整 Agent Runtime 和外部系统连接器。
