**整体建议：HOLD。**

## 三、P0 阻断问题

### P0-01 XCDOS 软删除定义会直接返回已删除数据

**证据**

- `XCDOS_数据库设计文档_DB_Design_V1.0.html:468` 规定默认查询条件为 `deleted_at IS NOT NULL`。
- 正常数据的正确过滤条件应为 `deleted_at IS NULL`。
- 同文档多处将 `created_at / updated_at / deleted_at` 合并为必填且默认 `NOW()`，例如第 108、122 行，容易被实现为新数据创建时即标记删除。

**影响**

列表、权限查询和统计可能只返回已删除数据，或所有新数据均处于删除状态。

**整改**

拆分三个字段定义；`deleted_at` 必须 nullable、默认 `NULL`；统一生成可执行 DDL，并增加软删除集成测试。

### P0-02 XCDOS 决策到执行链路缺少 Plan 完整契约

**证据**

- API 文档没有创建、提交、审批 Plan 的端点。
- `POST /api/tasks` 却要求 `planId` 必填。
- LLD 的决策报告允许 `plan` 为 null，却直接把案例状态改为 `resolved`。
- 产品验收要求“已审批方案 -> 创建任务”，当前 API 无法完成该流程。

**影响**

“决策报告 -> 方案审批 -> 任务执行”主链路无法通过公开 API 实现。

**整改**

补齐 `POST /plans`、`POST /plans/{id}/submit`、`POST /plans/{id}/approve|reject`、`GET /plans/{id}`；明确计划状态机、审批权限、审计和任务创建前置条件。

### P0-03 Forecast 有领域对象和功能要求，但无持久化模型

**证据**

- ARD 将 `Forecast` 定义为决策域对象。
- API 提供 `/decision-cases/:id/forecast`。
- TDD/DB Design 没有 forecast/scenario 表或明确 JSONB 存储契约。
- LLD 生成报告时未读取任何推演结果。

**影响**

“推”阶段结果无法可靠保存、复查、审计或进入最终报告。

**整改**

确定 `forecasts`、`forecast_scenarios` 模型，或明确聚合内 JSONB 结构；增加版本、输入假设、模型来源、人工修订、置信度和审计字段。

### P0-04 反馈不可修改与覆盖更新相互冲突

**证据**

- PRD/TDD/ARD 要求已提交反馈不可删除或修改，并保留历史。
- API、LLD 和验收用例允许同日重复提交时覆盖已有反馈。
- 数据库无反馈版本表、修订号、作废状态或变更原因。
- LLD 注释称“owner 或协作者可提交”，代码仅允许 owner。

**影响**

审计要求无法满足，历史反馈可被静默改写；前后端权限也会出现分歧。

**整改**

选择唯一规则：推荐“首次提交生成记录，后续修改生成 revision，不覆盖原文”；增加 `feedback_revisions` 或版本字段及唯一约束。

### P0-05 需求追踪矩阵已失真

**证据**

- `XCDOS_需求清单与需求基线_V1.0.html:262-272` 使用了不存在或不同的接口：
  - `/goals/{id}/children`
  - `/hypotheses`
  - `/evaluations`
  - `/roi-simulations`
  - `/feedbacks`
  - `/dashboard/kpis`
  - `/dashboard/exceptions`
- 实际 API 使用聚合端点，如 `/api/goals/:id/breakdown`、`/api/tasks/:id/feedback`、`/api/dashboard/boss`。
- 25 个功能项只追踪了 12 项。

**影响**

研发、测试和验收无法通过矩阵确认需求是否已实现。

**整改**

以功能 ID 为主键重建全量 RTM，覆盖 25/25 功能及非功能需求，并由 CI 校验 OpenAPI operationId、测试 ID 和需求 ID。

### P0-06 测试依赖未定义的接口与机制

**证据**

- `XCDOS_测试用例文档_V1.0.html:187` 使用未定义的 `DELETE /api/goals/:id`。
- 第 191 行使用未定义的 `/api/agent-runs/:id/cancel`。
- 第 192 行要求死信队列，但 BullMQ 队列设计未定义 DLQ。
- 第 186 行要求乐观锁返回 409，但任务表没有 `version` 字段，API 也没有 `If-Match`/版本参数。

**影响**

P0/P1 用例无法执行，测试准出结论不可信。

**整改**

先补 API、表结构和队列契约，再保留对应测试；否则删除超出范围的用例并登记后续版本。

### P0-07 Prolog 使用 MD5+盐保存密码

**证据**

- 总体方案和数据库设计均规定 MD5+盐。
- `Prolog ... DB Design V1.2.md:115` 明确 `password` 使用 MD5+盐。
- 联合安全文档又规定 bcrypt cost=12。

**影响**

安全基线自相矛盾，MD5 不适合密码存储，不能作为投产方案。

**整改**

统一为 Argon2id 或 bcrypt；定义参数、迁移策略、密码重置、失败锁定和历史密码策略。

### P0-08 Prolog API 与数据库核心契约分裂

**证据**

- API 的隔离模式为 `schema/shared`，数据库为“物理独立库/共享行级”。
- API 的规则类型为 `single/composite/fallback`，数据库为“流程/校验/路由”。
- API 的规则状态为 `draft/active/inactive`，数据库包含“未生效/已生效/灰度中”。
- API 的 `parent_rule_id` 为 UUID，数据库主键为 bigint 雪花 ID。

**影响**

前端、后端和数据库无法使用同一 DTO、枚举和迁移脚本。

**整改**

建立唯一领域字典；API、数据库、测试、规则规范全部引用同一枚举与 ID 类型。

### P0-09 Prolog “同时支持 MySQL/PostgreSQL”不具备可执行设计

**证据**

- `Prolog ... DB Design V1.2.md:22` 同时声明 MySQL 8/PostgreSQL 14+。
- 字段使用 `datetime`、`tinyint`、`ON UPDATE CURRENT_TIMESTAMP`、UTF8MB4、InnoDB 等 MySQL 方言。
- 部署、监控、备份又混用 MySQL binlog 与 PostgreSQL 工具。

**影响**

无法生成单一 DDL、迁移脚本、索引方案和运维手册。

**整改**

V1.2 必须选择一个主数据库。若确需双数据库，拆成两套 DDL、迁移、DAO 方言、测试矩阵和运维手册。

### P0-10 联合运维/安全文档未覆盖 Prolog 技术栈

**证据**

- Prolog 技术栈为 Spring Boot/Spring Cloud、Vue3、Quartz、MySQL/PostgreSQL。
- 联合运维手册仅提供 Next.js、NestJS、npm、TypeORM、PostgreSQL、BullMQ 和 XCDOS 镜像命令。
- 联合监控文档仅给出 `pg_stat_*` 和 BullMQ 指标。
- 联合安全文档以 React、TypeORM/Prisma、PostgreSQL 为实现前提。

**影响**

Prolog 实际部署、迁移、监控、回滚和安全扫描没有可执行 SOP。

**整改**

拆分 `XCDOS` 与 `Prolog` 两套运维/安全/监控/上线清单；共享部分只保留组织流程和统一控制要求。

### P0-11 Prolog 缺少被所有技术文档引用的 PRD V1.2

**证据**

总体方案、数据库设计和需求基线均声明前置依赖《PRD V1.2》，仓库中未发现该文件。

**影响**

无法证明功能、字段、价值、验收标准来自已批准需求，也无法完成需求追踪。

**整改**

补充并评审 PRD，或明确将需求基线升级为唯一需求源并修正文档引用。

### P0-12 数据合规闭环缺失

**证据**

- Prolog 规定核心业务数据永久保存、仅逻辑删除。
- 请求日志保存完整请求、响应、IP，并可作为训练样本。
- 未定义告知同意、最小必要、查阅复制、更正、删除、撤回、导出、委托处理、跨境传输和个人信息影响评估流程。
- 面向公众提供生成式 AI 服务时，还需评估生成式 AI 服务规范、投诉举报和用户输入记录保护要求。

**影响**

“满足金融、政务高合规场景”的结论没有证据，且永久留存与个人删除请求可能直接冲突。

**整改**

补充数据清单、处理目的、合法性基础、保留期限、删除/匿名化流程、数据主体请求 SOP、训练样本授权、供应商和跨境评估。合规结论须由法务/安全负责人签署。

## 五、P2 完整性与质量问题

1. PRD 计划为 8 周，TDD 为 8-10 周，ARD 将 Agent Runtime 排到 9-12 周，范围与排期未统一。
2. XCDOS PRD 的反馈目标为 `>=95%`，监控文档正常范围为 `>=80%`，管理目标和告警基线混淆。
3. 边界测试中多处使用“允许 A 或 B”“返回 null 或无穷”等非确定预期，无法自动化判定。
4. API 文档缺少 OpenAPI/JSON Schema，枚举、必填项和错误码只能人工同步。
5. Prolog 测试使用 `/api/v1.2/auth/current-user`，API 实际为 `/api/v1/auth/me`。
6. Prolog 第三方签名、防重放测试存在，但 API 文档没有 AppKey、签名串、Nonce 和时间戳协议。
7. Prolog 规则创建测试期望默认启用并生成快照，API 期望默认 `draft`，数据库也未明确首次创建快照时机。
8. 数据库“脱敏存储”与“业务查询不受影响”冲突。掩码数据不可逆，应区分加密存储和展示脱敏。
9. PostgreSQL 社区版并非直接提供文档所述通用 TDE 能力，需要明确云厂商、磁盘加密或扩展方案。
10. XCDOS 的 `roles` 等表未完全遵守“所有表统一四个审计字段”的声明。
11. XCDOS API 的 `DUPLICATE_FEEDBACK` 同时定义为 409 和“已自动更新”，语义冲突。
12. 运维手册使用 `redis-cli KEYS dashboard:*`，生产环境应改为 SCAN 或可观测索引。
13. “DB 连接耗尽 -> 重启 Pod”只能临时缓解，缺少连接泄漏、池配置和慢 SQL 根因 SOP。
14. 监控文档缺少指标计算 SQL/PromQL、持续窗口、缺失数据处理和仪表盘所有者。
15. 埋点将 `user_id`、`tenant_id`、`session_id` 与页面行为全量关联，但未定义权限、留存和匿名化。

## 七、建议整改顺序

### Gate 0：两天内完成基线止血

1. 修复 XCDOS 软删除条件和字段定义。
2. 停止使用 MD5+盐，统一密码安全方案。
3. 决定 XCDOS ORM、Prolog 主数据库和 ID 类型。
4. 补齐 Prolog PRD 或指定唯一需求源。
5. 冻结统一枚举字典、状态机和权限矩阵。

### Gate 1：一周内完成开发契约

1. 补齐 Plan/Forecast/反馈版本/Agent 取消等核心模型和 API。
2. 生成 OpenAPI、PostgreSQL DDL 或 Prolog 主数据库 DDL。
3. 重建完整 RTM。
4. 修正测试用例中不存在或模糊的预期。
5. 增加事务 Outbox、幂等、乐观锁和 DLQ 设计。

### Gate 2：开发中期完成安全与数据治理

1. 完成数据清单、日志脱敏、数据生命周期和删除流程。
2. 完成租户隔离实现方案与专项测试。
3. 完成 Agent 权限策略和自进化质量门。
4. 拆分双项目安全、监控和运维手册。

### Gate 3：上线前完成证据化准入

1. 压测报告支撑 QPS、P95/P99 和容量结论。
2. 故障演练支撑 RTO/RPO 和高可用结论。
3. 安全测试、恢复演练、灰度和回滚演练全部留档。
4. P0/P1 测试通过，并由产品、研发、测试、安全、SRE 签字。

## 八、Go / No-Go 门禁

### 允许进入 Sprint 0 的条件

- 产品负责人确认真实组织、账号、审批、反馈制度和试点部门。
- 技术负责人确认 XCDOS ORM、Prolog 数据库和部署边界。
- P0-01、P0-07、P0-08、P0-09 已修订。

### 允许进入全量开发的条件

- 12 个 P0 问题全部关闭。
- RTM 覆盖率达到 100%。
- OpenAPI 和 DDL 可由工具校验。
- P0 测试用例不存在未定义接口或非确定预期。

### 允许进入生产发布的条件

- 安全、合规、性能、容灾证据齐全。
- 高风险遗留项有明确接受人、到期日和补偿控制。
- 不再使用“完全满足”“彻底规避”“零丢失”“百万级”等无测试证据的绝对表述。

## 九、最终建议

项目方向和主业务链路值得继续，当前最大问题不是“缺文档”，而是文档数量已经超过契约治理能力。下一阶段应停止继续横向新增说明文档，先建立三个唯一事实源：

1. **需求事实源**：功能 ID、业务规则、验收标准。
2. **技术事实源**：OpenAPI、DDL、状态机、权限矩阵。
3. **交付事实源**：RTM、自动化测试、准入报告。

完成上述收敛后，XCDOS 可进入受控开发；Prolog 需完成数据库与多租户架构收敛后再解除 HOLD。
