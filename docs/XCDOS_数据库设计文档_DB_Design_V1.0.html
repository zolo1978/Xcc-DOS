XCDOS 数据库设计文档
DB Design V1.0

基于 TDD V1.0 第六章与 ARD V2.0 第五章，输出完整可落地的数据库设计。涵盖选型、17 表完整结构、索引策略、数据字典、生命周期管理、脱敏加密规则、缓存联动方案。

数据库：PostgreSQL 16 字符集：UTF-8 / utf8mb4 引擎：InnoDB (如用 MySQL) ORM：TypeORM / Prisma

一、数据库选型与基础配置

配置项选型理由
数据库系统PostgreSQL 16事务可靠、JSONB 灵活、pgvector 扩展、生态成熟
字符集UTF-8全中文支持，统一编码
Collationzh_CN.UTF-8中文排序规则
时区Asia/Shanghai (UTC+8)业务统一北京时间
ORMTypeORM / Prisma (待定)NestJS 生态首选
迁移工具TypeORM Migration / Prisma Migrate版本化数据库变更
连接池pgBouncer (生产) / 内置池 (开发)控制连接数，防止打满
分库分表MVP 不做（模块化单体）二期按租户/时间分片
MVP 采用单库多 Schema（public），不做物理分库。连接池设 20-50。所有表统一 4 个审计字段：id (UUID 主键), created_at (必填默认 NOW()), updated_at (必填默认 NOW())，deleted_at (nullable 默认 NULL，删除时写入时间戳；正常数据恒为 NULL，默认查询条件 deleted_at IS NULL)。

五、数据生命周期管理

数据类别保留周期清理策略归档规则
domain_events90 天定时任务每日清理 90 天前事件可选：归档至 ClickHouse（二期）
audit_logs1 年月度归档，超过 1 年物理删除归档至对象存储（JSON 格式）
agent_runs / tool_calls180 天季度清理 180 天前记录可选：保留摘要，删除详情
feedbacks2 年超过 2 年，保留统计摘要后删除明细年度归档
goals / tasks / plans永久软删除 (deleted_at)，不物理删除已删除数据保留 3 年后评估清理
organizations / users / roles永久仅逻辑删除，不物理删除不归档
所有清理操作必须在业务低峰期（凌晨 2:00-4:00）执行，清理前备份，清理后验证。不允许在生产高峰时段执行 DELETE / TRUNCATE。

六、数据脱敏与加密规则

表敏感字段保护方式规则

传输加密

  • 所有 API 强制 HTTPS/TLS 1.3
  • 数据库连接强制 TLS
  • Redis 连接强制 TLS (生产环境)

存储加密

  • PostgreSQL 透明数据加密 (TDE) — 生产环境推荐
  • 应用层加密：phone 字段 AES-256-GCM 加密存储

七、缓存与读写分离

7.1 缓存策略

缓存对象Redis Key 模式TTL更新策略
用户会话/Tokenauth:token:{user_id}24h登录写入，登出删除
角色权限auth:role:{role_id}1h权限变更时主动失效
驾驶舱 KPIdashboard:kpi:{org_id}5min定时刷新 / 事件驱动更新
目标进度goal:progress:{goal_id}10min任务/反馈变更时失效
Agent Run 结果agent:result:{run_id}30min运行完成写入，过期自动清除
组织树org:tree30min组织变更时主动刷新

7.2 缓存防护

  • 缓存穿透：布隆过滤器 + 空值缓存 (TTL 60s)
  • 缓存击穿：热点 Key 互斥锁 (SETNX)
  • 缓存雪崩：TTL 随机化（±20% 抖动）+ 多级缓存

7.3 读写分离

  • MVP 阶段：单主库，读写不分离
  • 二期引入：主库写 + 只读副本读（pgPool-II 或应用层路由）
  • 读写分离规则：写操作走主库，报表查询/驾驶舱聚合查询走只读副本
