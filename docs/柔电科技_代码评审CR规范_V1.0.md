# 柔电科技 代码评审（CR）规范 V1.0

## 一、CR 流程

```
开发提交 PR → 自动 CI (lint + test + build) 通过
  → 分配 Reviewer（≥ 1 人）
  → Reviewer 评审（3 个工作日内）
  → 有问题 → 开发修改 → 重新评审
  → 通过 → Reviewer Approve → 开发 Merge
```

## 二、评审维度 Checklist

### 功能正确性
- [ ] 代码实现了需求/Issue 描述的全部功能
- [ ] 边界条件处理正确（空值、0、负数、超长字符串）
- [ ] 错误处理完整（不吞异常，用户友好的错误信息）
- [ ] 状态流转逻辑正确（符合状态机定义）

### 安全性
- [ ] 无硬编码密码/API Key/Token
- [ ] 用户输入已校验（class-validator / Zod）
- [ ] SQL 使用参数化查询（无拼接）
- [ ] 权限检查在 Controller 层有 @Auth() 守卫
- [ ] 无敏感信息输出到日志

### 性能
- [ ] 无 N+1 查询
- [ ] 列表查询有分页
- [ ] 热点数据有缓存策略
- [ ] 无不必要的全表扫描

### 可读性
- [ ] 命名清晰、符合团队规范
- [ ] 函数 ≤ 50 行，文件 ≤ 400 行
- [ ] 复杂逻辑有注释
- [ ] 无注释掉的死代码

### 测试
- [ ] 有新功能的单元测试
- [ ] 测试覆盖核心逻辑 + 边界条件
- [ ] CI 中测试全部通过

## 三、评审意见分级

| 等级 | 含义 | 处理 |
|------|------|------|
| 🔴 Critical | 阻断性问题（安全漏洞、数据丢失风险、生产 crash） | 必须修复才能合并 |
| 🟠 Major | 严重问题（功能缺陷、性能隐患） | 建议修复，需给出理由才能绕过 |
| 🟡 Minor | 小问题（命名不规范、缺注释） | 建议修复，不阻塞 |
| 🔵 Suggestion | 改进建议 | 可选采纳 |

## 四、Reviewer 行为准则

- 对代码不对人，批评的是代码不是写代码的人
- 指出问题同时给出改进方案或示例
- 每条 Comment 标注等级（Critical/Major/Minor/Suggestion）
- 3 个工作日内完成首次评审

## 五、Author 行为准则

- PR 描述必须包含：变更摘要、测试说明、影响范围
- PR 粒度适中（≤ 400 行变更，大功能拆分为多个 PR）
- 对 Review Comment 逐一回应：修改 / 讨论 / 给出理由后标记 Resolved
- 所有 Comment Resolved 后主动通知 Reviewer 重新评审
